SkyHouse runs ads across Facebook, TikTok, Google, and Instagram for multiple clients simultaneously. Each platform sends conversion events via webhook (e.g., "purchase made," "lead submitted") in real time. We need a system that: (1) receives these webhooks reliably, (2) stores the data in a database, and (3) updates a live dashboard that clients can view. Walk us through how you would architect this system.


Tech stack — I would go for Node.js (Nest.js) + React.js + RabbitMQ + PostgreSQL + Redis, in a monorepo with TypeScript. Nest.js gives me structure and its guards/pipes are handy to validate and authenticate webhooks early. TypeScript across the monorepo lets me share the event types between gateway, worker and React. RabbitMQ because I need a work queue with per-platform routing, dead-letter and retry control, not a full event log (I would reconsider Kafka only if we need replay or much bigger throughput). PostgreSQL for consistency and querying, with jsonb for the raw payloads. Redis for caching aggregates and pub/sub.

Failures, retries, duplicates — I would split it in two: a gateway and a worker. The gateway only verifies the platform signature (HMAC, to avoid spoofed events), validates quickly, pushes to the queue and answers 2xx fast, so the platform does not re-fire. The worker processes the event. For deduplication I would not rely on the queue; I would use an idempotency key (from the platform event id) with a unique constraint in the database. For retries I would not do blind retries: a state machine where each failure state has its policy, watching attempts and the age of the event, and deprecating old ones; what keeps failing goes to a dead-letter queue with backoff. With many workers, I claim rows with SELECT ... FOR UPDATE SKIP LOCKED.

Scaling — Three directions. The gateway and worker are stateless, so I run more replicas and autoscale workers by queue depth. For more platforms, an adapter pattern: one adapter per platform plus a normalization layer mapping everything into a single canonical event, so a new platform is just a new adapter. For more clients, multitenant with tenant_id and Postgres row-level security; stronger isolation (schema or database per tenant) only when really needed.

Database — Two layers. A raw, append-only events table: ids, tenant_id, platform, idempotency key (unique index), received and expire timestamps, status, and a jsonb for the payload (hard to keep strictly typed). A second layer for the dashboard: aggregated tables or materialized views per client/platform/time bucket, because it can not scan raw jsonb on every load. It is time-series, so I partition by time and index on (tenant_id, platform, received_at). I would stay on PostgreSQL — jsonb already gives the flexibility, and it scales horizontally with replicas and partitioning when needed.

Dashboard — Since it must be live, I would use SSE instead of polling (simpler than WebSockets, reconnects by itself), fanning out with Redis pub/sub, and pushing aggregated deltas, not every raw event. I would keep a manual refresh and an interval option only as fallback.
